# uns 設計 — セッション引き継ぎメモ（2026-06-07）

前回メモ（2026-06-06、`unsnarl 0.5.0-beta.0`、読み取り専用が恒久確定）の続き。本セッションは「役割アンカー `@` の検討」から始まり、最終的に「位置クエリ層 vs 意味（観点）クエリ層」という構造的な整理に到達した。次のステップは「意味（観点）層の設計」。

---

## 0. 最優先の制約（前回から継続・不変）

- uns は読み取り専用。TS の生成・既存ファイルの書き換えは追加しない（恒久）。codemod は永久欠番。
- 出力可: 図・データ・指標・計画アーティファクト（`-o`/`--out-file` 含む）。出力不可: TS ソース・源泉の mutation。
- CLI は読み取り専用フィルタ族（`uns path -r nnn | pbcopy` の文化を維持）。

## 1. このセッションで確定したこと（再提案しない）

### 1-A. 役割記号 `@` と方向記号 `+` は併存させる（記号統一しない）

- 役割（種別選択）は `@`（例 `@param` / `@return` / `@import`）、方向は既存の `+a/+b/+c` のまま。
- beta ゆえ方向を `@a/@b/@c` に統一する破壊的変更も検討したが **却下**。
- 根拠は審美でなく **パーサ構造**（`crates/unsnarl-root-query/src/parse_root_query_ast.rs`）:
  - `..` の左辺は常に `parse_endpoint_query`（L56）のみ。左辺に方向を解釈する経路が無い。
  - 右辺は `rhs_text.starts_with('+')`（L58）の **先頭 1 文字・文脈自由** で方向/端点を即決。
  - → `+` は右辺限定の演算子、`@`（役割）は両辺可の端点。これは文法カテゴリの非対称であり、`@` 統一は L58 を壊して語彙依存のフォールバック解析を要求する。
- 設計原理: **「1 概念 = 1 記号」**（CSS セレクタ `#`/`.`/`:`、TS `#private` と同型。役割＝名詞／方向＝演算子の 2 概念だから記号も 2 つ）。

### 1-B. 方向トークンの記号選定史（揮発前に記録済み・issue #90 にも投稿）

1. 最初は正規表現 idiom `^`（先頭/源流）/ `$`（末尾/末端）を検討（`123..^` / `123..$`）。
2. 次に単記号 `@` / `+`（`123..@` / `123..+`）を検討。だが記号 2 つで方向を表すと「どっちが after（子孫）か」が記号から読めず詰まった。
3. `+a/+b/+c` に収束。記号は `+` に固定、方向は letter `a/b/c`（既存 `-A/-B/-C` と一致）が背負う。
- シェル実測: `^` は zsh `extendedglob` で否定グロブになり暴発（カレントディレクトリ展開）。先頭 `$` は全シェルで変数展開で消える。`@`/`+` は bash・zsh・extendedglob すべてでリテラル通過。
- **GitHub: issue #90 にクローズ後追記として投稿済み**（comment 4641034650）。

## 2. このセッションで探索し、疑問視/棄却したもの

### 2-A. `@param` / `@return` を「射影 `関数@return`」にする案 → 弱点が露呈

- 動機: `@return` が全関数の return に解決され「どの関数の return か」が一意でない問題への対処として、関数を基底にした射影（`a@return` = a の return）を検討。
- **破綻点（関数の一意指定が未解決）**:
  - 匿名関数は名前が無い → `123-456@return` のような行範囲基底が要る。
  - `() => () => v` のように 1 行複数アローだと行（範囲）でも一意にならない（実測: `g=(x)=>(y)=>x+y` は両アロー L3、scope_3/scope_4 で区別、内アローは `(anonymous)`）。
  - uns の出力に **column は無い**（`line`/`endLine` のみ）。一意に効くのは scope id か node ID 末尾のバイトオフセットだが、どちらも内部値でユーザのクエリトークンに不向き。

### 2-B. `..` への暗黙量化（per-function 自動グループ化）→ 論外として棄却

- `@param..@return` を「各関数の中で対応づけ」と暗黙に解釈する案は、分析ツールに隠れた振る舞いを入れることになり、`+`/`@` を分けてまで守った「意味はトークンで読める」原則に反する。**却下**。
- per-function 判定は `..` に隠さず、明示的に名前を持つ `-f` の計算（関数走査が宣言された契約）に置く、で合意。

## 3. 実機で確認した事実（grounded・再検証不要）

- **パターン A/B は公開バイナリの既存 `-H` トレースで手動検出が成立**:
  - パターン A（引数が return に寄与せず副作用に消える）: `a..+a` の塗りに return ノードが現れない。
  - パターン B（import が return に混入）: `7..+b`（return 行の上流）に `NamedImportBinding` が現れる。
- **役割に使える NodeKind は実在**（`crates/unsnarl-visual-graph/src/node_kind.rs`、全 28 種）:
  - source: `FormalParameter`/`CatchParameter`、`NamedImportBinding`/`DefaultImportBinding`/`NamespaceImportBinding`、`SyntheticImplicitGlobal`
  - sink: `ReturnArgumentReference`、`ThrowArgumentReference`、`WriteReference`、`SyntheticExpressionStatement`
- **中間束縛の通過**: `const`/`let`（再代入の `WriteReference` 経由を含む）は端から端まで辿れる。`var` のみ伝播が切れる（旧メモの「var を辿れない」は var キーワード固有）。
- **エッジラベルは read / set / call**（複合は `read,call`）。旧メモの「write」は実出力では `set`。
- **関数間到達は実在**（probe5: `setIt(a){shared=a}` の引数 a が module 可変 `let shared` 経由で `getIt(){return shared}` の return に到達）。ただしこれは「共有状態ハブ」の現れで、別の実ファイルを分析した際に観測した共有状態ハブと同一現象。
- ただし **`@return` 非一意の根本は多重性**（1 ファイル N 関数 → N return）であって共有の有無に依らない。
- uns は **return を最も近い named binding に帰属**させる（curry のネストは名前に畳まれ、`g@return` の射影先は named なら well-defined。途中の「外アローが内アローを返す」は return ノードにならない）。完全匿名インラインは一意ハンドルが無い。
- 既存文法（issue #90）: `query := point | path | direction`、`DIR_TOKEN := '+' [abc] [0-9]*`、`foo..bar` は方向非依存（到達集合の積）、経路なし → stderr 警告 + exit 0。

## 4. このセッションの到達点（区切りの整理）

### 4-A. 二軸の問い（`@params`/`@return` より前にあるべき問い）

設計は「`@params`/`@return` という記法」から始めるのではなく、ユーザの問いの順序から始める:

- **軸1（なにについて＝主体／スコープ）**: ファイル全体 / named な関数 / anonymous な関数。
  - 「一意指定」の難しさは全部この軸1 に閉じる。
  - **主体をカテゴリで選び（各関数／named 全部／anonymous 全部／ファイル全体）、ツールが内部で scope を走査する**なら、anonymous の一意指定問題は消える（人間が個別ハンドルを打たない）。
- **軸2（その主体の中で何を見るか＝側面）**: 一部を見る / input からの流れ（下流＝`+a`）/ output に至る流れ（上流＝`+b`）。
  - input/output は **主体から導出される境界**（関数の input = その params、output = その return と副作用）。宙に浮いた `@param`/`@return` ではない。
  - だから軸1 を先に答えた時点で「どの関数の return か」は構造的に発生しない。

検算: パターン A = 「主体の input から下流に流して output に届かない」、パターン B = 「主体の output から上流に遡ると import に行き着く」。どちらも〈軸1 で主体〉→〈軸2 で向き〉の二問で言い切れる。

### 4-B. メタな診断（このセッションの核心）

**unsnarl の任務が「グラフを描画する」から「観点に基づいて可視補助する」に進化したことで、既存クエリが軋み始めている。**

- 既存クエリ（`-r` / `-A・-B・-C` / `-H` / `--depth`）は **「グラフ要素を位置で選ぶ」層**。名前・行で node を指し、近傍を広げ、経路を塗る。語彙は位置と地形。「描画」には過不足ない。
- 新任務は **「意味で選ぶ」層**。params・return・input・output・主体・向き＝役割と流れの意味で問う。
- 軋みの正体: **意味的な意図を位置語彙に通そうとして折れている**。最近詰まった四つは全て同じ症状:
  1. 「どの関数の return か」= 意味的主体に位置的住所が無い。
  2. 暗黙量化の誘惑 = 意味は「各関数について」と量化、位置クエリは一領域選択のみ。
  3. 否定・不在（パターン A）= 意味は「届かない」を含む、位置の塗りは presence のみ。
  4. 二軸（主体/側面）= 意味層に構造があるのに、平らな位置クエリは表せない。
- **向かう先**: 位置クエリ文法を継ぎ接ぎするのではなく、**観点を上の層に置き、それが内部で位置クエリへコンパイルされる**形。「各 named 関数についてパターン A」→ ツールが関数を列挙 → 各 input/output 境界を導出 → 既存の到達クエリを発行して塗る。人間は観点を述べ、ツールが位置に翻訳する。位置層（`+`/`@` 分割・パーサ規律）は無傷で残る。
- 4-A の二軸（主体／側面）と、合意済みの「明示的 `-f` 内部走査」は、まさにこの **上層（意味/観点層）の文法**。

---

## 5. 次のステップ（区切りの先）

- **意味（観点）層の設計**: 軸1（主体の選び方）× 軸2（側面＝流れの向き）を、どういう表面（`-f` の分析名・フラグ等）で人間に出すか。位置層へどうコンパイルするか。
- **否定・不在（パターン A）**: 「経路なし」報告／塗られ残りの差分として、この上層で扱う。位置層に否定演算子は足さない。
- **anonymous の指し方**: カテゴリ選択＋内部走査で溶ける見込み。確証を取る。
- 出力形態の決定（既存 `-H` のハイライトを上層が生成するのか、`-f` の独立アーティファクトか）。

## 6. 行き止まり（再提案しないこと）

- 方向トークンの `@a/@b/@c` 統一（パーサ構造で却下）。
- `@param`/`@return` を「ユーザが関数を名指しする射影」にする案（匿名・同一行で一意指定が破綻）。
- `..` への暗黙量化（分析ツールに隠れた振る舞い、論外）。
- （前回メモから継続）型/モジュール/effect の結合エッジ追加。metrics/partition/verdict を「足りない機能」として追加する方向。
