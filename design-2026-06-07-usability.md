# unsnarl 使い勝手向上に関する設計討議 議事録

- 日付: 2026-06-07
- 議題: unsnarl をより使いやすくするための方向性
- 位置づけ: 設計・概念レベルの討議。実装内部は本議事録の対象外（監修対象は設計と概念のみ）。
- 状態: 未決のまま、ユーザー判断で打ち切り（末尾参照）。

---

## 1. 出発点の問い

対話シェルモード（ターミナル上のツール）と VSC 拡張のどちらを作るか。

## 2. 消費者の再定義（重要な転換）

- 当初の前提: 出力は視覚的（Mermaid）であり、ボトルネックは人間の視覚処理 → VSC 拡張が有利に見えた。
- 転換: 出力の消費者は LLM かもしれない。Mermaid は人間の目にしか効かず、LLM にはノイズ。LLM には plain text（解決済みのスコープ／参照情報）の方が効く、という仮説。
- 帰結: 消費者は「人間の目（→ Mermaid）」と「LLM（→ テキスト）」の二種に分かれる。判断軸が「誰が消費するか」「誰がナビゲートするか」に整理された。

## 3. 本当の痛点（作者＝エンドユーザー視点）

- ビュー絞り込みのオプションを手で組むのが面倒。
- エディタで名前や行番号を見て、頭に記憶し、シェルに書き写すのが面倒。
- 出力を手動でコピーして Claude Code に渡し、さらに「解析依頼」プロンプトを書く、という連鎖がソリューションとして弱い。
- 整理: **人間が「エディタ / unsnarl / Claude Code」の糊（手作業の連結役）をやらされている**。
  - Leg A（入力側）: 選択対象の転記。
  - Leg B（出力側）: 結果の運搬。

## 4. 対話ツールの正体（認識のすり合わせ）

- 想定が食い違っていた。コマンドを手打ちする REPL ではなく、**構造駆動の TUI ナビゲータ**。
- ファイルパスを渡すと、パース結果から選択肢（imports / functions / トップレベル）が提示され、関数に潜ると input / statements / return … と辿っていける。
- 候補を提示するのは unsnarl 側なので、エディタからの転記（Leg A）が起点から消える。
- 留意: Leg B（出力の運搬）はこのツール単体では消えない。確定アクションを Claude Code への直渡しにする設計が必要。

## 5. CLI + JSON 出力 と MCP の差

- ローカル・shell 利用可・単一ファイルという前提では、両者の終着点（JSON が LLM の文脈に入る）はほぼ同じ。
- MCP 固有の強み: (1) shell を叩けない環境への到達性、(2) 発見性＋構造化された入力、(3) リモート／クロスプラットフォーム提供、(4) 狭い権限、(5) ステートフルなセッション（実利は小）。
- MCP 固有ではないもの: 「JSON を文脈に入れること」自体、出力の構造化。

## 6. 設定渡し（-c）の位置づけ

- MCP の「構造化入力」の利点は、サーバを建てずに設定渡しで取れる。
- 一方「到達性」側は設定渡しでは取れない。
- ただし到達性は本ユースケース（自分の Mac ＋ Claude Code ＋ shell 利用可）ではほぼ不要。よって設定渡しで MCP を当面後回しにできる。
- 設定渡しは対話ツールの内部表現に留まらず、単独で価値を持つ（LLM が自分で設定を書ける／将来 MCP を建てる際の入力契約になる）。

## 7. ロードマップは「在／不在」でなく「優先順位」

- 全項目はロードマップ上に在る。問いは順序と着手の引き金。
- 能動的に作る対象: **設定渡し** と **対話ツール** の 2 点。
- 後回し（引き金待ち）:
  - MCP（unsnarl-server）— 引き金: 到達性が現実に必要になったとき。
  - VSC 拡張 — 引き金: 人間向けビジュアル需要が実際に立ったとき。
- 未検証: LLM 向けの compact 出力が必要か。既存の出力で足りるかを、実ファイルを Claude Code に渡す安い実験で判定する（※当初これを最優先項目として提示したが、根拠不十分として撤回。要否そのものが未確定）。

## 8. IR の位置づけ（重要な論点修正）

- IR の冗長さは難点ではない。**疎結合を保つための防波堤**。
- 「IR で出せる」という制約そのものに価値があり、消費者の有無で測る対象ではない。
- 機能追加は消費者駆動で議論してよい（重要）。ただしその過程で IR を下げる場面は来ない。IR は機能軸のトレードオフに乗らない不動点。compact 表現が要るとしても、それは IR の下流の射影として足すものであり、IR を削る方向ではない。

## 9. 設定渡しと対話ツールの関係

- producer / consumer の関係。対話ツールが設定を産み、unsnarl と LLM が消費する。
- 要件は疎結合。「どちらを先に作っても詰まない」こと。
- それを保証するのは、両者が互いを参照せず、先に在る共通の契約（既存の「どのスライスを見せるか」の指定の仕組み）だけを向くこと。
- これは OCP ＋ Strategy であり、unsnarl は既にこの設計。新しい抽象を起こす必要はない。
  - （討議中、実装寄りの寄り道で「新しい型の切り出しが必要」と述べたが、共通契約は既に存在しており不要、として撤回。）

## 10. 残る唯一の設計判断（未決）

対話ツールの「歩き方」（情報設計）。これは実装ではなく、人が構造をどう辿るかの設計:

1. 各階層で何を見せるか（粒度）。最上段は imports / functions / その他トップレベル。関数配下は input / statements / return、およびその関数の呼び出し元・読み書き・呼び出し相手。
2. 項目を選んだときの意味。単一に絞るか、複数を選んで焦点の集合を積むか。
3. 歩き終えた後の確定アクション。表示するだけか、Claude Code へ直渡しするか。

- 要確認: imports / functions の区別が既存データから取れるか（グルーピングが無償か否かを分ける）。
